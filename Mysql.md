### 1.5 表的键（key）

1. 主键 (Primary Key) —— “唯一的身份证”

这是最重要的一种键。**一张表必须有，且只能有一个主键。**

- **特点：**
  1. **唯一性：** 任何两行数据的主键值都不能相同（不能有两个学生叫 1 号）。
  2. **非空性：** 主键不能为空（NULL）。
  3. **速度快：** 数据库会默认给主键建立最高效的索引，查主键是最快的。



2.  外键 (Foreign Key) —— “连接的桥梁”

   这是实现**多表关联**的核心。**外键是另一张表的主键**。

   - **作用：** 它用来建立两张表之间的联系，并保证数据的**逻辑正确性**（引用完整性）。
   - **规则：** 如果你在“学生表”里填了一个班级 ID 是 99，但“班级表”里根本没有 99 号班级，数据库会**报错**，不让你存。这就防止了脏数据的产生。



3. 唯一键 (Unique Key) —— “虽然不是老大，但也独一无二”

它和主键很像，要求列里的数据不能重复。

- **区别：**
  - **地位不同：** 主键是“老大”，只能有一个；唯一键是“VIP”，一张表可以有很多个（比如手机号要唯一、邮箱也要唯一）。
  - **空值：** 主键不能是 NULL；但在 MySQL 中，唯一键允许是 NULL（除非你特别指定不能为 NULL）。

### 2.3 聚合函数

1. 基础用法

聚合函数的功能：将**某一列的多行数据**作为输入，经过计算后，返回**单个值**作为输出。这通常用于统计数据，比如计算总和、平均值或统计数量，而不是简单地列出数据。

（所有的`NULL`不参与聚合函数的运算）

| **函数名**  | **含义** |
| ----------- | -------- |
| **COUNT()** | 计数     |
| **SUM()**   | 求和     |
| **AVG()**   | 平均值   |
| **MAX()**   | 最大值   |
| **MIN()**   | 最小值   |

例如有张成绩表`scores`：

| **name** | **subject** | **score** |
| -------- | ----------- | --------- |
| 张三     | 数学        | 90        |
| 李四     | 数学        | 80        |
| 张三     | 英语        | 70        |
| 李四     | 英语        | 60        |

查询所有人的平均成绩：

```mysql
select AVG(score) from scores
```

输出结果：75



### 2.4 分组查询

1. 基础语法

```mysql
select 字段列表 from 表名 group by 分组字段列表 having 分组后过滤条件
```

核心作用：把一张大表里的数据，按照你指定的规则，切分成很多个“小堆”。

例：有一个 `students` 表：

| **id** | **name (姓名)** | **class (班级)** | **score (分数)** |
| ------ | --------------- | ---------------- | ---------------- |
| 1      | 小明            | **一班**         | 90               |
| 2      | 小红            | **二班**         | 85               |
| 3      | 小强            | **一班**         | 70               |
| 4      | 小刚            | **二班**         | 60               |
| 5      | 小李            | **三班**         | 80               |

执行分组查询语句

```mysql
SELECT class FROM students GROUP BY class;
```

查询结果：

| **class** |
| --------- |
| 一班      |
| 二班      |
| 三班      |

>  单独使用 `GROUP BY` 而不配合聚合函数时，效果有点像“去重”（DISTINCT），它只是列出了所有的类别。类似于excel中的`筛选`下拉菜单，列出一共有哪几种值
>
>  分组查询时，`select`后面的字段列表和`group by`后面的字段列表要保持一致（除了聚合函数），否则会导致行数不统一而报错



2. ⭐分组查询+聚合函数

单纯的分组通常没有意义，我们分组是为了**统计每一组的特征**。这就是聚合函数出场的时候。

结合的逻辑：先分组，再对每一组进行统计计算

对于表 `students`，我们使用下列查询语句：

```mysql
SELECT class, AVG(score) 
FROM students 
GROUP BY class;
```

>  此时数据库的查询逻辑和步骤为：
>
> 1. **拆分：** 先按照 `class` 把大表拆成三个“小表”（一班表、二班表、三班表）。
>    - 一班小表：包含小明(90)、小强(70)
>    - 二班小表：包含小红(85)、小刚(60)
>    - 三班小表：包含小李(80)
> 2. **计算：** 对**每个小表**分别执行聚合函数 `AVG(score)`。
>    - 一班计算：(90+70)/2 = 8
>    - 二班计算：(85+60)/2 = 72.5
>    - 三班计算：80/1 = 80
> 3. **合并：** 把计算结果拼成一张新表返回给你。

查询结果：

| **class** | **AVG(score)** |
| --------- | -------------- |
| 一班      | 80             |
| 二班      | 72.5           |
| 三班      | 80             |

3. 分组后的查询`having`

```mysql
select 字段 from 表名 where 分组前过滤条件 group by 分组字段名 having 分组后过滤条件
```

`where`和`having`的区别：

1. 执行时机不同：where在分组前查询，不满足where条件的不会参与分组；having是在分组后
2. 判断条件不同：where不能对聚合函数进行判断，而having可以
3. 执行顺序：`where`>聚合函数>`having`

例如

```mysql
select class_name,avg(score) 
from students 
where gender = '女' 
group by class_name 
having avg(score) > 90;
```

含义：给出女生平均分大于90分的班级及其女生平均分

- where gender = '女'：导致男生不参与分组
- having avg(score) > 90：只列出（女生）平均分大于90的扳机



### 2.5 排序查询

关键字：`order by`

基础语法

```mysql
select 字段列表 from 表名 order by 字段1 排序方式1,字段2 排序方式2……
```

两种排序方式

- `ASC`：升序（默认，不需要额外写）
- `DESC`：降序



### 2.6 分页查询

关键字：`limit`

作用：用来**限制查询结果返回行数**

语法：

```mysql
select 字段列表 from 表名 limit 起始行数,查询行数
```

解释：

- 起始行数：表示**跳过**多少行（从 0 开始计数）。
- 查询行数：表示每页**显示**多少行。



### 2.7 查询语句执行顺序

一个最全查询语句长这样：

```mysql
SELECT 字段列表
FROM 表名列表
WHERE 条件列表
GROUP BY 分组字段列表
HAVING 分组后条件列表
ORDER BY 排序字段列表
LIMIT 分页参数
```

每个部分的执行顺序为：

1. From：确定在哪个表里查
2. where：筛选条件，不满足条件的不参与后续步骤
3. group by：分组
4. having：对分组结果再进行一次筛选
5. select：开始查询
6. order by 和limit by：这两个只影响最终查询结果呈现的形式，不影响实质内容

### 3.4 流程函数

1. IF函数（条件判断函数）

语法：`IF(条件, 成立时的值, 不成立时的值)`

例：快速把 GPA 大于等于 2.0 的标记为“及格”，否则标记为“挂科”。

```mysql
select name,if(gpa>=2.0,'及格','挂科')
from students;
```



2. IFNULL函数（处理空值`null`）

语法：`IFNULL(value, 默认值)`

- 如果value是 `NULL`，就返回你指定的默认值。
- 如果value**不是** `NULL`，就返回value原本的值。



3. CASE WHEN函数（多条件判断）

语法：

```mysql
CASE 
    WHEN 条件1 THEN 结果1
    WHEN 条件2 THEN 结果2
    ...
    ELSE 兜底结果
END
```

例：根据 GPA 给学生评级：

- GPA >= 3.8 → **优秀**
- GPA >= 3.0 → **良好**
- GPA >= 2.0 → **及格**
- 其他 → **不及格**

```mysql
SELECT 
    name, 
    gpa,
    CASE 
        WHEN gpa >= 3.8 THEN '优秀'
        WHEN gpa >= 3.0 THEN '良好'
        WHEN gpa >= 2.0 THEN '及格'
        ELSE '不及格'
    END AS level
FROM students;
```



4. 流程函数+ `group by`

例：统计“优秀、良好、及格、不及格”这四个档次各有多少人

```mysql
select
    case
        when gpa >= 3.8 then '优秀'
        when gpa >= 3.0 then '良好'
        when gpa >= 2.0 then '优秀'
        else '不及格'
    end as level,
    count(*)
from
    students
group by level;
```

思路：

1. 先用 `CASE WHEN` 给每个人打上标签（level）
2. 再把这个标签当作分组依据。

> 这里可以用`as`为分组依据起一个别名`level`，这样group by的时候就无需再写一遍`case when`
>
> 标志的sql语句是不可以这样用的，因为group by的执行顺序在select前面，MySQL对这里进行了**特殊优化**，导致可以这样用



## 第四讲 多表查询

### 4.1 多表关系

1. 一对多（多对一）关系

这是数据库中最普遍的关系，也是我们刚才创建的 `students` 和 `classes` 之间的关系。

- 定义： A 表中的**一条**数据，对应 B 表中的**多条**数据。但反过来，B 表中的一条数据，只能对应 A 表中的一条。
- 例子：**班级 vs 学生：** 一个班级有**很多**个学生；但一个学生只能属于**一个**班级。
- 实现方式： 我们通常在**“多”**的那一方（学生表），加一列**外键**（`class_id`），指向**“一”**的那一方（班级表）的主键。



2. 多对多关系

- 定义： A 表的一条数据对应 B 表的多条；B 表的一条也对应 A 表的多条。
- 例子：**学生 vs 课程**。 一个学生可以选择**很多**门课（数学、英语）；一门课也可以被**很多**学生选。
- 实现方式：引入第三张表，叫做**“中间表”**（或者关联表）。



3. 一对一关系

定义： A 表的一条数据，**严格**只对应 B 表的一条数据。

例子：

- **用户 vs 身份证信息：** 一个用户只能有一个详细的身份证档案。

> **为什么不写在一张表里？** 通常是为了**性能**或**安全**。
>
> - 比如 `users` 表只存常用的账号密码（查得快）。
> - 另一张 `user_details` 表存身份证号、家庭住址、健康档案等（查得少，且敏感）。



### 4.2 多表查询分类

1. 笛卡尔积

如果直接执行下列查询语句：

```mysql
SELECT * FROM students, classes;
```

数据库会做一个最笨的操作：**它会拿表 A 的每一行，去和表 B 的每一行都配对一次。**

- 学生1 对应 班级1
- 学生1 对应 班级2
- 学生1 对应 班级3
- ...
- 学生15 对应 班级3

**结果总行数 = 表 A 行数 × 表 B 行数：**$$15 \times 3 = 45 \text{ 行}$$

这就是 **笛卡尔积**。

在多表查询中，我们要消除笛卡尔积，例如可以使用下列语句：

```mysql
SELECT * FROM students, classes where students.class_id = classes.class_id;
```

即可把数据不匹配无用行（笛卡尔积行）都删去

### 4.3 连接查询

连接查询是最常用的查询方式，它的直观理解是：**把两张表（或多张表）横向拼在一起，变成一张更大的宽表，然后再利用匹配条件把笛卡尔积消除掉**

1. 连接查询进一步细分类：

| **连接类型** | **关键字**          | **核心逻辑** |
| ------------ | ------------------- | ------------ |
| **内连接**   | `JOIN..ON`或`WHERE` | **取交集**   |
| **左连接**   | `LEFT JOIN..ON`     | **左边全留** |
| **右连接**   | `RIGHT JOIN..ON`    | **右边全留** |

内连接语法：

```mysql
select 字段列表 FROM 表1, 表2 where 连接条件
```

或

```mysql
select 字段列表 FROM 表1（左表） JOIN 表2（右表） ON 连接条件
```

外连接：把`JOIN`换成`LEFT JOIN`或者`RIGHT JOIN`



### 4.4 自查询

通常我们做 JOIN 是把“表A”和“表B”连起来。而自连接是把**“表A”**和**“表A”**连起来。

>  为什么要自己连自己？

这种查询通常用于处理**“层级关系”**或**“树状结构”**的数据。

最经典的场景是：

- **员工表：** 员工是你，你的上司也是员工，你们都在同一张表里。我想查“员工的名字”和“他上司的名字”。
- **网页表：** 网页 A 有一个链接指向 网页 B，它们都在“网页表”里。

> 既然是同一张表，数据库怎么区分哪一行是“员工”，哪一行是“上司”呢？

我们需要给这张表起两个不同的**别名 (Alias)**，把它想象成**两张完全独立的表**。

- **表 1 (s1)：** 我们把它看作 **“学生表”**。
- **表 2 (s2)：** 我们把它看作 **“组长表”**（其实还是学生表，只是角色变了）。

例：查询每个学生的姓名及其组长的姓名

```mysql
select s.name as '学生姓名', l.name as '组长姓名'
from students as s join students as l on s.leader_id = l.student_id;
```



### 4.5 联合查询

**联合查询 (UNION)** 是 SQL 中另一种非常重要的多表操作方式。

如果说 JOIN (连接查询) 是把两张表 **“横向”** 拼在一起（增加列），那么 **UNION (联合查询)** 就是把两张表 **“纵向”** 堆叠在一起（增加行）。

| **关键字**    | **功能**            | **性能**                | **举例**                                                    |
| ------------- | ------------------- | ----------------------- | ----------------------------------------------------------- |
| **UNION**     | **合并 + 去重**     | **慢** (因为要检查重复) | 如果学生叫“张三”，老师也叫“张三”，结果只显示**1个**“张三”。 |
| **UNION ALL** | **合并 + 保留重复** | **快** (直接堆叠)       | 学生“张三”和老师“张三”都会显示，**共2行**。                 |

语法：

```mysql
-- 第一部分：查学生
SELECT name AS 姓名, '学生' AS 身份 FROM students

UNION ALL  -- 核心关键字：把上下两部分粘起来

-- 第二部分：查老师
SELECT teacher_name, '老师' FROM classes;
```

 UNION 的两大规则

使用 `UNION` 必须遵守两个死板的规定，否则报错：

1. **列数必须相同：** 上面的查询查了两列，下面的查询也必须查两列。你不能上面查“姓名+年龄”，下面只查“姓名”。
2. **字段类型要兼容：** 上面查的是日期，下面对应的那一列最好也是日期（或者能转成日期）。你不能把“生日”和“价格”堆在一列里。



### 4.6 子查询

核心思想：嵌套`select`语句；在一个 SQL 语句（主查询）的内部，再嵌套一个完整的 SQL 语句

根据**子查询结果**（即嵌套内的select结果）的不同，子查询可以分为以下种类：

| 种类       | 子查询结果         |
| ---------- | ------------------ |
| 标量子查询 | 单个值（一行一列） |
| 列子查询   | 一列               |
| 行子查询   | 一行               |
| 表子查询   | 一个表（多行多列） |

1. 标量子查询

例：找出 **“比平均绩点（GPA）高的学生”**。

思路：先计算平均绩点，再查询

```mysql
select
    name,gpa
from students
where
    gpa > (select avg(gpa) from students);
```

括号内的就是子查询，他计算出了平均绩点



2. 列子查询

例：找出 **“所有在 '数据一班' 或 '艺术班' 的学生”**。

难点：`students`表中只有班级的`class_id`，没有班级名称，所以先找出`class_id`对应的班级名称，再找出其中的学生

列子查询中还涉及一些逻辑运算符来配合筛选：

| **逻辑运算符** | **配合符号** | **逻辑翻译**         |
| -------------- | ------------ | -------------------- |
| **IN**         |              | 等于其中任意一个     |
| **NOT IN**     |              | 不等于列表里任何一个 |
| **ANY**        | `>`          | **大于最小值**       |
| **ANY**        | `<`          | **小于最大值**       |
| **ALL**        | `>`          | **大于最大值**       |
| **ALL**        | `<`          | **小于最小值**       |

```mysql
SELECT name, class_id 
FROM students 
WHERE class_id IN (
    SELECT class_id 
    FROM classes 
    WHERE class_name IN ('数据一班', '艺术班') -- 子查询返回一个列表 (1, 3)
);
```

列子查询的缺陷：

子查询在 `WHERE` 子句中使用时最大的**局限性**：**“由于作用域（Scope）隔离，主查询无法访问子查询内部的字段。”**

> 如果只用列子查询，最终的查询结果只能给出students表中的数据，得不到子表中的数据
>
> 即：
>
> ```mysql
> select s.name, c.class_name
> 
> from students as s
> 
> where
> 
> s.class_id IN (select c.class_id 
>                from classes as c 
>                where c.class_name in ('数据一班','艺术班'));
> ```
>
> 
>
> `select s.name, c.class_name`这里会报错，因为不能识别classes表的别名c

行子查询和列子查询的思想类似



3. 表子查询

子查询的结果是**多行多列**，可以直接把它**当成一张“临时表”**放在 `FROM` 后面，并且必须为其起一个别名

例：查询“GPA > 3.5 的学生中，女生的学生信息

```mysql
select *
from (select *
      from students
      where gpa > 3.5
     ) as h
where h.gender = '女';
```



4. 关联子查询⭐

前面的子查询都是“先执行里面，再执行外面”，里面和外面是独立的。 但在关联子查询中，**里面的查询依赖外面的数据**。

例：找出每个班级里，比该班级平均分高的学生

```mysql
select name, gpa, class_id,avg_gpa.平均绩点
from students
join (select class_id as '班级',avg(gpa) as '平均绩点' from students group by class_id) as avg_gpa
on class_id = avg_gpa.班级 and gpa > avg_gpa.平均绩点;
```



## 第五讲 窗口函数

### 5.1 概述

> 为什么要用窗口函数？

例：查出每个学生的姓名、班级、成绩，以及他所在班级的平均分

传统的 `GROUP BY` 做不到，因为 `GROUP BY` 把多行压缩成了一行。即只显示班级的平均分，无法显示每个班级下每个同学的具体情况。

窗口函数的优势：

在不减少行数（不折叠数据）的情况下，给每一行数据“贴”上一个统计标签。

> 相当于给每一行都新增一列字段（平均分），例如运行以下查询语句：
>
> ```mysql
> SELECT
>     name,
>     class_id,
>     gpa,
>     -- 这就是窗口函数：算出平均分，但不折叠行
>     AVG(gpa) OVER (
>         PARTITION BY class_id
>     ) AS class_avg_gpa
> FROM students;
> ```
>
> 这个语句会为每一行都加一个名为`class_avg_gpa`的字段，同时不损失任何信息



### 5.2 窗口函数语法

1. 语法：

```
<函数名> OVER (
    PARTITION BY <列名>   -- 1. 分区 (把谁归为一组？)
    ORDER BY <列名>       -- 2. 排序 (组内怎么排队？)
)
```

**关键词详解**

- **`OVER()`**：这是**“窗口”**的标志。看到它就说明是窗口函数。
- **`PARTITION BY`**：**“分区”**。
  - 类似 `GROUP BY`。
  - 比如 `PARTITION BY class_id`，就是把 1 班的放一个窗口，2 班的放一个窗口。计算平均分时，**互不干扰**。
- **`ORDER BY`**：**“组内排序”**。
  - 在窗口内部，数据是怎么排列的？（这一步对“排名”函数至关重要）。



### 5.3 常用窗口函数

1. 排名函数（ranking）

常用的排名函数有三种：

- **`ROW_NUMBER()`**：**点名法** (1, 2, 3, 4)。不管分数是否一样，强制排名。
- **`RANK()`**：**奥运金牌法** (1, 1, 3, 4)。两人并列第一，第三个人就是第三名（跳过第2名）。
- **`DENSE_RANK()`**：(1, 1, 2, 3)。两人并列第一，第三个人是第二名（不跳号）。

例：给每个班级的学生按 GPA 排名（班内排名）

```mysql
select * , rank() over(
    partition by class_id
    order by gpa
    ) as ranking
from students
```



2. 聚合函数

使用的就是之前的聚合函数，但是结合窗口函数，而不是`group by`

例：查看每个学生的分数，以及**他与班级平均分的差距**。

```mysql
select name, gpa, class_id, avg(gpa) over(
    partition by class_id
    ) as avg_gpa,
    gpa - avg(gpa) over(
    partition by class_id
    ) as diff
from students
```

> 注意：计算和平均分的差距`diff`时，不能写成`gpa - avg_gpa`，即利用之前的别名，而是要把`avg_gpa`的完整公式再写一遍
>
> 原因： SQL 的 `SELECT` 语句是**并行执行**的，而不是从左到右逐行执行。**数据库会告诉你：** “找不到列 `class_avg_gpa`”。因为在这一层查询里，这个别名还未“出生”。
>
> **所以，必须把计算逻辑 `AVG(gpa) OVER (PARTITION BY class_id)` 完整地再写一遍。**



3. 偏移函数（环比计算）

- **`LAG(col, n)`**：往前找第 n 行的数据。
- **`LEAD(col, n)`**：往后找第 n 行的数据。

例：对每个班级进行排名，并找出每个同学前1名的同学作为自己的“目标”（第一名没有）

```mysql
select name, gpa,class_id, lag(name,1) over (
    partition by class_id
    order by gpa
    ) as target
from students;
```



练习题：取出**每个班级**中，GPA 最高的 **前 2 名** 学生

```mysql
select name, gpa, ranking,class_id
from (
    select name, class_id,gpa,rank() over (
    partition by class_id
    order by gpa
    ) as ranking
    from students
     ) as new_ranked
where ranking<=2;
```

思路：先利用子查询生成每个班级的排名表`new_ranked`作为临时表，再在其中用`where`筛选排名`ranking<=2`

>  陷阱：**不能**直接在窗口函数后面写 `WHERE rank <= 2`。 因为 SQL 的执行顺序是：`FROM` -> `WHERE` -> `SELECT (窗口函数在这里执行)`。 也就是说，在筛选的时候，排名还没算出来，`ranking`这个字段都还没有定义



### 5.4 窗口范围控制

**什么是窗口范围控制？**

在前面的计算中，如果我们根据`partition by class_id`将所有数据分区以后，如果计算`avg`，那么函数计算的范围是整个分区（class）里所有的数据，但是有时候我们并不想要所有的数据，而是指定一个范围内的数据（例如从第一行到**当前一行**），这时就要用到窗口范围控制。

窗口函数的完整语法：

```mysql
<函数名> OVER (
    PARTITION BY <列名>
    ORDER BY <列名>
    [ROWS|RANGE] BETWEEN <开始位置> AND <结束位置>  -- 窗口范围控制
)
```

开始位置和结束位置的关键词：

- **`CURRENT ROW`**：当前这一行。
- **`n PRECEDING`**：往前数 n 行 (Before)。
- **`n FOLLOWING`**：往后数 n 行 (After)。
- **`UNBOUNDED PRECEDING`**：从最开始的第一行算起（起点）。
- **`UNBOUNDED FOLLOWING`**：一直算到最后一行（终点）。



例：按gpa从高到底排序，并算出**“截止到每一位学生时，平均gpa是多少”**。

```mysql
select name, gpa,avg(gpa) over(
    order by gpa
    rows between unbounded preceding and current row
    ) as current_avg_gpa
from students;
```



> `rows`和`range`的区别

`ROWS`按照**物理行**计算，只认行号，不管数据是什么样的

`RANGE` 是按**“逻辑值”**算： 它是**灵活的**。它看的是 `ORDER BY` 后面那个字段的**具体数值**。如果数值相同，它会把它们视为“同一堆”一起处理。



例：有一张销售订单记录表，现在想做这几天销售的**流水账**和**销售日报**

- 流水账：做**财务流水**，每一笔交易进来，余额都要变动一次（哪怕是同一秒的两笔交易），**必须用 `ROWS`**。
- 销售日报：只关心“截止到当天”的总量，不关心当天内部的先后顺序，**可以用 `RANGE`**。

用`rows`做流水账：

```mysql
select id,sale_date, sum(amount) over(
    order by sale_date
    rows between unbounded preceding and current row
    ) as sum_sale
from daily_sales;
```

用`range`做销售日报：
```mysql
select id,sale_date, sum(amount) over(
    order by sale_date
    range between unbounded preceding and current row
    ) as sum_sale
from daily_sales;
```

结果对比：

| **sale_date**  | **金额** | **ROWS 结果 (逐行加)** | **RANGE 结果 (按天结)** | **此时发生了什么？**                           |
| -------------- | -------- | ---------------------- | ----------------------- | ---------------------------------------------- |
| 2024-01-01     | 100      | 100                    | 100                     | 第一天，没区别。                               |
| **2024-01-02** | 200      | **300** (100+200)      | **500** (100+200+200)   | **关键点！** RANGE 把这一天的两笔全加进去了。  |
| **2024-01-02** | 200      | **500** (300+200)      | **500** (同上)          | RANGE 这一行结果也是 500，因为它俩是“并列”的。 |
| 2024-01-03     | 300      | 800                    | 800                     | 恢复正常。                                     |
| 2024-01-04     | 100      | 900                    | 900                     | 恢复正常。                                     |





## 第六讲 CTE和视图View

### 6.1 CTE复杂查询

1. 什么是 CTE？

CTE 全称 **Common Table Expression**，中文叫“公用表表达式”。 你可以把它想象成编程里的 **“定义临时变量”**。

- **以前（子查询）：** 你必须把所有逻辑写在一个巨大的 SQL 里，像剥洋葱一样一层套一层。
- **现在（CTE）：** 你可以先定义一个“临时表 A”，再定义一个“临时表 B”，最后再从 B 里面查数据。

CTE语法：

```mysql
WITH 临时表名 AS (
    -- 这里写你的子查询逻辑
    SELECT ...
)
SELECT * FROM 临时表名;
```

**优点：** 

- 逻辑是“从上往下”读的。第一步干什么，第二步干什么，清清楚楚。
- 可以一次定义好几个，而且后面的可以引用前面的

还是以前的例子：取出每个班级前 2 名

使用cte语法：

```mysql
with ranked_table as (
    select name,class_id,gpa,rank() over (
        partition by class_id
        order by gpa
        ) as ranking
    from students
)

select *
from ranked_table
where ranking <=2;
```

注意：

- 在一个`with as`中定义多个临时表时，中间用逗号`,`隔开

- `with as`定义的临时表只在紧接着的那条 `SELECT`、`UPDATE` 或 `DELETE` 语句中有效

  > 如果你下次还想用这个逻辑，你得**把代码重写一遍**;（或者把它保存成我们后面要学的 **视图 View**）



### 6.2 视图View

如果说 **CTE** 是你写的一张“草稿纸”（用完就扔），那么 **视图 (View)** 就是你装订好的“教科书”（永久保存，随时翻阅）。

1. 什么是视图 (View)？

一句话概括：**视图是一张“虚拟表” (Virtual Table)。**

- **它不存数据：** 它的肚子里没有数据，它只存储一段 **SQL 代码**（也就是查询逻辑）。
- **它不是表：** 当你查询视图时，数据库会“即时”运行它里面保存的那段 SQL语句，算出结果给你看。



2. 为什么要用视图？

假设你每天都要给老板发一份“优秀学生名单”，逻辑是这样的：

1. 把 `students` 和 `classes` 表连起来 (JOIN)。
2. 筛选 GPA > 3.5 的学生。
3. 按分数排序。

**如果没有视图：** 你每天都要写一遍这坨代码（或者复制粘贴）。

 **有了视图：** 你把这坨代码存成一个视图叫 `honor_students`。以后每天只需要写 `SELECT * FROM honor_students` 就可以了，大大减少重复工作。



3. 如何创建和使用视图

创建视图 ：在你平时写的查询语句前面加一行：

```mysql
-- 创建一个视图，名字叫 honor_students
CREATE VIEW honor_students AS

-- 下面是你熟悉的查询逻辑 (核心代码)
SELECT 
    s.name AS 学生姓名, 
    c.class_name AS 班级, 
    s.gpa AS 绩点
FROM students s
JOIN classes c ON s.class_id = c.class_id
WHERE s.gpa >= 3.5;
```

一个视图只能对应“一个查询结果集”。

常用写法：用 `WITH` 定义多个临时的 `SELECT`，最后输出一个。

使用视图：就像查询普通表一样查询即可

```mysql
-- 以后老板要名单，你只需要：
SELECT * FROM honor_students;

-- 你甚至可以对视图再进行过滤：
SELECT * FROM honor_students WHERE 班级 = '数据一班';
```

| **特性**     | **CTE (Common Table Expression)** | **View (视图)**                       |
| ------------ | --------------------------------- | ------------------------------------- |
| **持久性**   | **临时的** (语句跑完就没了)       | **永久的** (存在数据库里，除非你删它) |
| **存储位置** | 内存/临时空间                     | 数据库字典 (Schema)                   |
| **用途**     | 用来搞定“一次性”的复杂逻辑拆解    | 用来搞定“重复性”的高频逻辑封装        |
| **能否复用** | 只能在当前查询里复用              | 任何查询、任何人都能复用              |

## 第七讲 索引

**什么是索引？**

利用B+树这种数据结构，把查询的“逐个遍历”转化为“层层查询”，以空间换时间，大幅提升查询速度1. 

**索引的种类**

1. 主键索引：**主键 (Primary Key)** 本身就是一种最高级的索引

2. 普通索引：需要手动创建，语法为
   ```mysql
   CREATE INDEX 索引名 ON 表名(列名);
   ```

3. 唯一索引：在`唯一`列上创建，要求该列不能有重复值（例如身份证）
   ```mysql
   CREATE UNIQUE INDEX 索引名 ON 表名(列名);
   ```

4. 组合索引：为多个列创建同一个索引（比如经常查询 `WHERE age = 25 AND username = 'Alice'`）
   ```mysql
   CREATE INDEX idx_age_name ON users(age, username);
   ```



**索引相关操作**

1. 删除索引
   ```mysql
   DROP INDEX 索引名 ON 表名
   ```

2. 查看索引
   ```mysql
   SHOW INDEX FROM 表名;
   ```



**什么时候 不需要 建索引**

由于索引需要占用空间，所以不能任意创建索引，以下情况不要创建：

- 表的数据量太少：比如就10行数据，没有索引也查询的很快

- 频繁更新的列： 比如“最后登录时间”、“访问次数”。每次改这个数字，数据库都要去重新排一下索引树，性能损耗巨大。

- 区分度低的列： 比如“性别”。

  > 如果你建了索引，目录里一半是“男”，一半是“女”。你要找“男”，数据库通过目录发现还是得要把这一半的数据都看一遍。**这种索引完全失效，不如全表扫描。**



**验证索引是否生效**

命令关键字：`EXPLAIN`

**操作：** 在你的 `SELECT` 语句前面加一个 `EXPLAIN`。

```mysql
-- 假设我们给 username 加了索引
EXPLAIN SELECT * FROM users WHERE username = 'Alice';
```

看结果中的 `type` 列：

- **`ALL`**：**全表扫描**。最差的情况！说明没用到索引（索引失效），正在一页页翻书。
- **`ref`**： **非唯一索引扫描**。还不错，通过索引找到了。
- **`const`**： **常量级查找**。最快！通常是主键或唯一索引，一下就定位到了。



**索引失效**

建立索引并不等于“强制”数据库去用它。MySQL数据库中的查询优化器 (Cost-Based Optimizer, CBO)在执行查询语句前，会先算一笔账：

- **方案 A（走索引）：** 查目录 -> 拿到 ID -> 回去查正文 -> 也就是“跳着查”。
- **方案 B（全表扫描）：** 不看目录 -> 直接从第一行读到最后一行 -> 也就是“顺着查”。

如果 MySQL 算出方案 B 的“成本”更低（速度更快），它就会抛弃索引，也就是索引失效了。

索引失效的情况总结：

| **场景**     | **错误写法 (索引失效)** | **正确/推荐写法**         | **原因**             |
| ------------ | ----------------------- | ------------------------- | -------------------- |
| **联合索引** | `WHERE b=1` (跳过头)    | `WHERE a=1 AND b=1`       | 违背最左前缀法则     |
| **列运算**   | `WHERE age+1=20`        | `WHERE age=19`            | 破坏索引值的有序性   |
| **函数操作** | `WHERE YEAR(date)=2024` | `WHERE date BETWEEN...`   | 相当于对列进行了重写 |
| **类型转换** | `phone=123` (String列)  | `phone='123'`             | 触发隐式类型转换函数 |
| **模糊查询** | `LIKE '%abc'`           | `LIKE 'abc%'`             | 前缀未知，无法定位   |
| **OR连接**   | 一边有索引，一边没有    | 两边都有索引 或用 `UNION` | 木桶效应，取决于短板 |

> **MySQL 的索引（B+树）是按照顺序排列的**。所有的失效情况，本质上都是因为你的 SQL 写法破坏了这种“有序性”，导致数据库无法利用原本排好序的结构来快速定位数据。
>
> 详细解释如下：
>
> 1. 核心比喻：B+树 = 只有底层存书的“多级图书馆目录”
>
> 想象你走进了一个巨大的图书馆，书架上摆放着几百万本书（数据行）。为了让你快速找到书，图书馆建立了一套特殊的**索引系统（B+树）**。
>
> 这个系统有三个关键特点，这正是 B+树的精髓：
>
> #### 特点一：数据全在最底层（叶子节点）
>
> B+树是一棵“矮胖”的树。
>
> - **树顶和树腰（非叶子节点）：** 这里**不放书（数据）**，只放**路牌（索引键值）**。比如写着“1-500号去左边，501-1000号去右边”。它们的作用只有一个：**指路**。
> - **树底（叶子节点）：** 所有的**书（真实数据或指向数据的指针）**都安安静静地躺在这里。
>
> > **为什么要这样？** 因为路牌很小，书很厚。如果不放书，每一层就能放更多的路牌，树就会变得更“矮”。就算有几千万本书，这棵树通常也只有 3 到 4 层高。这意味着你只需要翻 3-4 次牌子就能找到目标，速度极快。
>
> #### 特点二：严格的从左到右排序（有序性）
>
> 这是最最重要的一点！
>
> 不管是在树顶指路，还是在树底放书，所有的号码都是**严格按照从小到大顺序排列的**。
>
> - 左边的肯定比右边的小。
> - 第 1 本书肯定是 ID=1，第 2 本是 ID=2……
>
> #### 特点三：底部手拉手（双向链表）
>
> 在最底层的叶子节点（存放书的地方），所有的架子都是**手拉手连在一起的**。
>
> - 找到了第 50 号书，不用回到树顶，直接往右伸手就能拿到第 51 号、52 号……
> - 这使得**范围查询**（比如“找 50 到 100 之间的书”）非常快。
>
> ------
>
> 2. 用 B+树解释为什么索引会失效？
>
> 现在脑海里有了这棵“有序的、多层指路、底部连通”的树，我们再来看那些失效场景，你会发现它们简直是“理所当然”的。
>
> #### 场景 A：为什么 `LIKE '%华'` (模糊查询在前) 会失效？
>
> - **B+树视角：** 你的索引树是按照拼音 `A-Z` 或者笔画严格排序的。
>   - 如果你搜 `LIKE '刘%'`（姓刘）：管理员看树顶，知道“刘”在“李”和“马”之间，瞬间定位。
>   - 如果你搜 `LIKE '%华'`（名字带华）：管理员傻眼了。“刘德华”在 L 区，“任达华”在 R 区，“霍建华”在 H 区。
> - **结论：** 因为开头不确定，**路牌（排序）失效了**，管理员只能把图书馆几百万本书一本本拿出来看名字里有没有“华”。这就是**全表扫描**。
>
> #### 场景 B：为什么 `WHERE age + 1 = 20` (计算) 会失效？
>
> - **B+树视角：** 树是按照 `age` 的原始值（18, 19, 20...）排序好的。
>   - 你现在问：“谁的年龄加 1 等于 20？”
>   - 树说：“我只排了 `age` 的序，没排 `age+1` 的序啊！虽然我知道 `age=19` 在哪，但你让我算 `age+1`，我就得把所有人拉出来，每个人头顶上加个 1 算一遍，再看等不等于 20。”
> - **结论：** 一旦对索引列做运算，原本的有序结构就无法直接对应了。
>
> #### 场景 C：为什么联合索引 `(a, b)`，只查 `b` 会失效？
>
> - **B+树视角：** 联合索引就像是**按照“班级-学号”排序**的档案。
>   - 先按班级排（1班、2班...），如果班级一样，再按学号排。
>   - **情况 1（查班级 `a`）：** 既然先按班级排的，当然能直接找到“1班”在哪。
>   - **情况 2（查班级 `a` 和学号 `b`）：** 先找“1班”，再在“1班”里找“5号”，非常顺畅。
>   - **情况 3（只查学号 `b`）：** 你问“谁是 5 号？”。管理员看了一眼档案：1班有个5号，2班也有个5号，3班还有... 它们在物理上是分散的，不是连在一起的。
> - **结论：** 跨过了老大（a），老二（b）在全局上是无序的（只有在 a 确定的**局部里才有序**）。这就是**最左前缀法则**的由来。
>
> #### 场景 D：为什么类型转换（字符串 vs 数字）会失效？
>
> - **B+树视角：** 字符串的排序规则和数字完全不同。
>   - **数字排序：** 2, 10, 20
>   - **字符串排序：** "10", "2", "20" (因为 '1' 比 '2' 靠前)。
> - **结论：** 如果字段是字符串索引，你传个数字进来，MySQL 必须把每一行字符串转换成数字才能比较，这又回到了“场景 B”的运算问题，破坏了原有的排序逻辑。
